## Invariant Rules

1. **Module scope**: One module per task — don't cross boundaries without explicit plan.
2. **No silent failures**: If a criterion cannot be met, report it — never skip.
3. **No suppression**: Never use suppression comments/flags (`# noqa`, `@ts-ignore`, `--no-verify`, or the equivalent for this stack) to pass gates.
4. **Atomic commits**: Each task gets its own commit with `feat|fix|test|docs|chore` prefix.
5. **Security first**: No secrets in code, no auth bypass, no CSRF/CSRF-equivalent skip.
6. **Thin coordinator**: Coordinator dispatches agents → agents do the work. Coordinator does NOT perform codebase analysis, code review, gate execution, or AC verification directly. Exception: executor prompt assembly (Phase 1.5) is coordinator's responsibility.
7. **MCP pre-checks mandatory**: Every executor prompt (initial AND corrective) MUST include Step 0 (specialist knowledge) and Step 1 (MCP pre-checks, using this product's configured code-discovery MCP) BEFORE Step 2 (implementation). No exceptions.
8. **Progress tracking**: Update the state-dir phase marker at each phase transition. Use cross-session memory search (if a memory plugin like claude-mem is configured in this project) for prior-context retrieval.
9. **Browser-verification for frontend, if the capability exists**: If contour = frontend/full-stack/ui-design AND this product has a configured browser-verification tool (e.g. Playwright) for its frontend profile:
   - a browser-verification agent (e.g. `accessibility-tester`) runs in Phase 2.2 and Phase 4.2
   - screenshots — only as files, paths in the report; do NOT embed images in the LLM context
   - if the tool is NOT configured for this product, this step degrades to a documented skip (`no-browser-tool-configured`) — that is not a silent skip, it must be logged as such. If the tool IS configured, skipping it is a blocker.
10. **Frontend design workflow, if the product has one**: For frontend/full-stack stories, Phase 1.4 loads whatever design-contract mechanism this product's frontend profile provides (see `references/design-modes.md` — this runtime does not ship one; it is an optional frontend-profile capability). If the product has no such mechanism, Phase 1.4 degrades to a lightweight design review against `.claude/agents/ui-designer.md` (if present) — skipping Phase 1.4 entirely for a frontend story is still a blocker.
11. **Backend MCP checks**: For backend/full-stack stories:
    - context7 — `resolve-library-id` + `query-docs` for EVERY framework/library actually used — MANDATORY regardless of stack.
    - this product's DB-verification MCP (if one is configured, e.g. `postgres_ro`) — MANDATORY before writing/changing SQL, migrations, or ORM models when the story touches a DB.
    - Skipping context7 for a backend story, or skipping DB-verification when a DB-verification MCP IS configured and the story touches the DB, is a blocker.
12. **Phase 0 mem-search BLOCKING, if a memory plugin is configured**: Phase 0.1 (mem-search) MUST run before Phase 1 when this project has a cross-session memory plugin. Without it, the worker loses prior-work context and burns budget re-discovering it. If the plugin is unavailable at call time, record "phase-0-failed" and continue with a warning — do not silently skip. If no such plugin is configured in this project at all, this invariant does not apply.
13. **Phase 5.3 cross-session summary BLOCKING, if a memory plugin is configured**: Phase 5.3 (structured summary observation) MUST run before closing the story when a memory plugin is configured. Without it, future sessions lose context of this work. A story closed without the summary is an incomplete delivery.
14. **Phase 5.1 single-writer SRS contract, gated by `srs.enabled`**: when the SRS/RTM module is enabled, the worker NEVER edits the active SRS or its archive doc directly. The worker only updates story frontmatter (`status: done`) and writes `${CLAUDE_WORKER_DIR}/srs-pending.json` with the proposed status/archive-move/notes per `requirement_ref`; RTM is coordinator-single-writer too (regenerated post-merge, never hand-committed by the worker). When `srs.enabled: false`, the whole SRS workflow doesn't apply — but the worker still writes `srs_pending: true` (as a no-op placeholder) into `close.json`, because the phase-contract schema requires one of `{srs_pending, srs_updated}` to be true regardless of whether the module is on. Full algorithm — `references/srs-sync.md`.
15. **Phase 2.1 parallel dispatch HARD-MANDATORY**: tasks from `plan.json.tasks[]` with empty/satisfied `depends_on` MUST be dispatched in ONE assistant-message with N `Agent(subagent_type="story-executor")` tool_use blocks — NOT a series of N separate messages. Symptom of violation: in the Claude Code TUI the first agent sits in foreground, the second queues; the user has to manually background the first for the second to start. Correct behavior: one message → multiple `Agent()` blocks → the harness runs them concurrently → automatic multi-agent fanout. Serial dispatch is allowed ONLY for a real code-dependency (`task-B.depends_on: [task-A.id]` and A is not done) or `files_affected` overlap. Full algorithm + anti-patterns — `references/phase-2-implement.md` §2.1.1.
16. **CWD discipline in the Bash tool**: ALWAYS use absolute paths. `cd` in the Bash tool is discouraged — shell-state sticks (one failed `cd` from a failed `eval` breaks every subsequent command with pathspec errors). Use an absolute path directly (`bash /abs/path/to/scripts/ao/X.sh`) or `bash -c 'cd /abs/path && cmd'` (subshell, not loose-state). If you accidentally end up in a subdirectory (`pwd` doesn't show the worktree root) — immediately `cd` back to the worktree root (`ao.worktree_root`/`STORY-{ID}`) to recover.
